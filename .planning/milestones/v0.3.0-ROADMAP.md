# Milestone v0.3.0: In-Cluster Execution

**Status:** ✅ SHIPPED 2026-04-20

**Total Plans:** 13 plans across 7 phases

## Overview

Enable k6 test execution inside the Kubernetes cluster via k6-operator, with ConfigMap script sourcing and metric extraction from runner pod logs. Extends the plugin beyond Grafana Cloud by adding a second provider that drives k6-operator TestRun CRDs, with full end-to-end validation on a kind cluster.

### Phase 08.2: Default cfg.Parallelism=1 when unset (INSERTED)

**Goal**: Bug fix: `buildTestRun` forwarded `cfg.Parallelism` as `int32(0)` when the user omitted `parallelism` in plugin config. k6-operator treats `spec.parallelism=0` as "paused" — no runner pods spawn and the AnalysisRun hangs forever. Fix defaults `cfg.Parallelism == 0` to `1` inside `buildTestRun` (pure builder, no mutation, no logging); `ValidateK6Operator` accepts `0` as "unset" unchanged.
**Depends on**: Phase 08.1
**Plans**: 1 plan

### Phase 08.3: Remove spec.Cleanup=post (INSERTED, retroactive)

**Goal**: Bug fix discovered via `/gsd-debug` after Phase 08.2 let TestRuns actually run to completion. `buildTestRun` set `spec.Cleanup = "post"`, and k6-operator v1.3.x calls `r.Delete(ctx, k6)` on the TestRun CR (cascade-deleting runner pods) as soon as stage transitions to `finished`/`error`. This race-deletes the CR before the metric plugin's 10s Resume poll can read the terminal stage and parse `handleSummary` from pod logs, producing `testruns.k6.io "..." not found` and driving the AnalysisRun to `phase=Error` after `consecutiveErrorLimit` retries.
**Depends on**: Phase 08.2
**Plans**: 1 plan

Plans:
- [x] 08.3-01-PLAN.md — Remove `spec.Cleanup=post` from `buildTestRun`; invert `TestBuildTestRun_Cleanup` into `TestBuildTestRun_CleanupUnset`; extend `dumpK6OperatorDiagnostics` with AR/Rollout yaml and controller log tails

**Verification:** `08.3-VERIFICATION.md` status `passed` (10/10 checks PASS). TestK6OperatorStepPass 62.12s PASS (was 326.84s FAIL), TestK6OperatorMetricPass 9.52s PASS (was 92.84s FAIL). Full `make test-e2e` 261s, 6/6 tests PASS, 0 FAIL.

## Milestone Summary

**Decimal Phases (all inserted post-Phase-10 as e2e bugs surfaced):**
- Phase 08.1 — metadata wiring (urgent, exposed by Phase 10 e2e)
- Phase 08.2 — parallelism default (urgent, exposed by re-running e2e after 08.1 fix)
- Phase 08.3 — spec.Cleanup=post removal (retroactive, discovered via `/gsd-debug` after 08.2 let TestRuns complete)

These three decimal phases form a debugging chain: each fix exposed the next hidden bug. 08.1 made the plugin's k6-operator provider reach k6-operator at all; 08.2 let the TestRuns actually run (previously paused); 08.3 stopped k6-operator from deleting the CRs before the plugin could observe their terminal state. The final e2e run after 08.3 showed 6/6 tests passing.

**Key Decisions:**
- Use dynamic client (unstructured) for TestRun CRD — avoid importing controller-runtime. (Good — kept dep tree small.)
- `client-go` v0.34.1 promoted from indirect to direct dependency.
- Pass/fail from runner pod exit codes (k6-operator issue #577 workaround).
- `handleSummary` JSON from pod logs for metric extraction — no sidecar, no extra CR.
- PluginConfig field defaults live at the consumer site (builder function), not in `parseConfig` or the validation layer. (08.2 decision — preserves "0 means unset" at boundary, keeps builders pure.)
- e2e regression guards read back emitted CRs via `kubectl jsonpath` rather than inferring from downstream phase — higher signal than Rollout-Healthy inference.
- Do NOT set `spec.Cleanup` on TestRun CRs. (08.3 — k6-operator's `post` semantics cascade-delete the CR before the plugin can observe terminal state. Success-path cleanup belongs in `GarbageCollect`/terminal hook — future phase.)
- Argo-rollouts controller logs are the authoritative source of plugin stderr when debugging (go-plugin pipes binary stderr into the controller's log stream). Any e2e diagnostic must capture those logs.

**Issues Resolved:**
- Namespace fallback to `default` bug (08.1) — no longer reproduces in e2e.
- Paused-TestRun hang (08.2) — runner pods now spawn reliably.
- TestRun-GC-before-Resume race (08.3) — metric plugin can observe terminal state; AnalysisRun reaches `Successful` cleanly.
- Diagnostic blind spot (08.3) — e2e failures now surface plugin stderr via controller logs, not just TestRun/pod dumps at teardown.

**Issues Deferred (Technical Debt):**
- Success-path TestRun/pod cleanup — currently leaks completed CRs in target namespaces. `StopRun` handles Terminate/Abort paths. A follow-up phase should implement `GarbageCollect` on the metric plugin and a symmetric post-terminal delete on the step plugin. Not urgent (resource accumulation, not functional break).
- Combined step+metric canary with mid-run AnalysisRun deletion — D-07 owner-ref precedence not exercised by e2e. Covered by unit tests; GC cascade is kube-apiserver behavior. Worth adding to e2e eventually.
- Three INFO-level items from 08.1 code review (ambiguous warning message on non-controller owner ref; missing nil-check defense on `rollout.Name`; owner-ref test doesn't lock Controller/BlockOwnerDeletion nil state). Intentional trade-offs; optional hardening.

**Technical Debt Incurred:**
- `internal/provider/operator/testrun.go` has two scopes of Godoc (the original plus the 08.2 parallelism explanation plus the 08.3 cleanup explanation) — readable but worth consolidating in a future sweep.
- `dumpK6OperatorDiagnostics` in the e2e test is now ~35 lines of kubectl shell-outs. Worth extracting to a helper if more dumps get added.

**Git stats:** ~99 commits scoped to v0.3.0 phases, 8,913 total Go LOC across `internal/`, `e2e/`, `cmd/` after milestone close.
