## Decisions Made

All decisions were pre-locked in `08.2-CONTEXT.md` (D-01 through D-08). Plan revision from REVIEWS.md locked the additional review-driven decisions:

- **Consumer-site defaulting (D-01):** The default lives inside `buildTestRun`, not in a `PluginConfig` helper. No `ResolvedParallelism()` added (no second caller exists); no `parseConfig` mutation (would break `TestValidateK6Operator_ZeroParallelism`'s "0 means unset" contract).
- **Silent default (D-02):** `buildTestRun` stays a pure builder. No slog emission. Operators verify via `kubectl get testrun -o yaml`.
- **Validation untouched (D-03):** `ValidateK6Operator` is byte-identical. The "0 means unset, default downstream" contract is preserved at the config boundary.
- **Enrich existing e2e tests, not duplicate (review MEDIUM-1):** Both fixtures (`rollout-step-k6op.yaml`, `analysistemplate-k6op.yaml`) already omit `parallelism`, so they already traverse the buggy path. Enriching them with direct CR-field assertions is higher-signal than adding a redundant `TestK6OperatorStepParallelismUnset`.
- **Direct CR readback (review MEDIUM-2):** e2e uses `kubectl get testruns -o jsonpath={.items[0].spec.parallelism}` to read the emitted value directly. Proves the plugin wrote `1`, not that the Rollout advanced for some other reason.
- **One-sentence README (review LOW-3 / D-08):** The note is a single sentence appended to an existing section, not a new `## Parallelism` subsection. No example YAML carries `parallelism: 1` -- the whole point of the default is users don't set it.
