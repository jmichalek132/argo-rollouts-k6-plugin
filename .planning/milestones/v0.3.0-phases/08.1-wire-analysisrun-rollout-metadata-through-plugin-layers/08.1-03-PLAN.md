<nil_rollout_note>
**R8 (from codex review):** Nil Rollout preserves pre-fix behavior (empty metadata, provider operates on user cfg + `default` fallback). This is graceful degradation, not a guarantee of correct provider behavior when metadata is absent — the provider-specific failure mode remains whatever it was before this fix. In particular, `stopActiveRun` will still call `StopRun` with an under-populated `cfg` if `rollout == nil`; whether StopRun can resolve the CR in that state is a provider-specific question out of scope for this bug fix.
</nil_rollout_note>

<objective>
Wire Rollout metadata through `internal/step/step.go` so `cfg.Namespace`, `cfg.RolloutName`, and `cfg.RolloutUID` are populated before provider dispatch in `Run`, `Terminate`, and `Abort`. This is symmetric with Plan 02 (metric plugin) but uses the Rollout parent directly — no owner-ref walk is needed because the Rollout IS the parent (D-06).

Purpose: Today all three step methods bind the Rollout argument to `_` and discard it. The shared `stopActiveRun` helper (used by Terminate and Abort) never sees the Rollout, so StopRun's cfg lacks the Namespace/RolloutName needed to resolve the TestRun CR name in the operator provider. D-05 adds `RolloutUID` (Plan 01 already landed the field); this plan populates it.

Output: `step.go` Run, Terminate, and Abort bind `rollout`; Terminate/Abort pass it through to `stopActiveRun`. A shared helper (`populateFromRollout`) applies D-01 namespace precedence and D-09 nil-Rollout warning. Three new tests with a `testRollout` fixture prove the plumbing: populated-cfg on Run, user-namespace-wins on Run, and **StopRun receives a populated cfg through Terminate** (R6 — closes the codex-identified verification gap on stop paths). All existing nil-Rollout tests keep passing.
</objective>
