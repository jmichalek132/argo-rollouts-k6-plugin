# Phase 08.1: Wire AnalysisRun/Rollout metadata through plugin layers - Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Bug fix: the metric plugin (`internal/metric/metric.go`) discards `*v1alpha1.AnalysisRun` and the step plugin (`internal/step/step.go`) discards `*v1alpha1.Rollout`, so `cfg.Namespace`, `cfg.RolloutName`, `cfg.AnalysisRunName`, and `cfg.AnalysisRunUID` are never populated before dispatching to the provider. Exposed by Phase 10 e2e tests — the `readScript()` fallback to `"default"` namespace causes ConfigMap lookups to miss in the real test namespace and the operator cannot construct correct TestRun owner references.

Phase 8 already designed for this plumbing (see D-07, D-09 in Phase 8 CONTEXT.md) but the metric/step layer never implemented it. This phase lands the extraction on both plugins, extends the TestRun owner-ref logic to accept a Rollout parent when there is no AnalysisRun, and adds targeted regression tests.

**Scope is confined to:** `internal/metric/metric.go`, `internal/step/step.go`, `internal/provider/config.go` (new `RolloutUID` field), `internal/provider/operator/operator.go` (owner-ref selection logic), and the corresponding `_test.go` files. No changes to routing, CRD handling, log parsing, or e2e test YAML.

</domain>

<decisions>
## Implementation Decisions

### Namespace Precedence & Fallback
- **D-01:** Namespace resolution order is **user cfg.Namespace > parent (AR/Rollout) namespace > `"default"` fallback**. The metric and step plugins populate `cfg.Namespace` from the parent object ONLY when the user left the config field empty. Preserves the Phase 8 D-07 intent (user can target a different namespace than the Rollout) while making the common path work automatically.
- **D-02:** The existing `"default"` fallback in `operator.go` `readScript()` (lines 145–148) **stays in place** as a defensive safety net. After this fix it becomes unreachable in normal plugin paths (metric/step always populate `cfg.Namespace`), but leaving it avoids silent breakage for any direct-provider callers and documents the precedence tail.

### Rollout Name Derivation (Metric Plugin)
- **D-03:** Metric plugin walks `ar.OwnerReferences` for `kind == "Rollout"`, preferring the entry with `Controller == true`. The Argo Rollouts controller stamps this reference when it creates the AnalysisRun (verified in argo-rollouts `rollout/analysis.go:502`: `*metav1.NewControllerRef(c.rollout, controllerKind)`). Authoritative and requires no label heuristics.
- **D-04:** If no Rollout owner reference is found (standalone AnalysisRun — e.g. e2e tests that kubectl-apply an inline AR, manual troubleshooting), log a warning and leave `cfg.RolloutName` empty. The operator already tolerates empty `RolloutName` in TestRun CR naming (existing code in `operator.go` constructs the name with whatever `cfg.RolloutName` holds). Do not fabricate placeholders.

### Step Plugin: Rollout Ownership
- **D-05:** Step plugin has no AnalysisRun, so `cfg.AnalysisRunName` and `cfg.AnalysisRunUID` stay empty in the step path. Instead, extend `PluginConfig` with a new field `RolloutUID string` (json tag `"-"`, same pattern as the other injected fields). `step.go` populates `cfg.RolloutUID = string(rollout.UID)`. Metric plugin does not set this field.
- **D-06:** `step.go` extracts metadata from the `*v1alpha1.Rollout` argument symmetrically with `metric.go`: `cfg.RolloutName = rollout.Name`, and `cfg.Namespace` is set to `rollout.Namespace` only when the user left `cfg.Namespace` empty (per D-01).
- **D-07:** `operator.go` owner-reference precedence when building TestRun CRs: **AnalysisRun wins when `cfg.AnalysisRunUID != ""`; else Rollout wins when `cfg.RolloutUID != ""`; else no owner reference.** The most-immediate parent gets the GC relationship. Metric plugin naturally selects AR; step plugin naturally selects Rollout. The existing `TestTriggerRun_WithoutAnalysisRunUID` test already asserts the "no owner ref" branch; a new test must assert the Rollout-owned branch.

### Error Handling & Regression Tests
- **D-08:** Add targeted new tests with non-nil AR/Rollout fixtures; **keep existing `nil` AR/Rollout tests as-is** — they exercise config parsing and don't need metadata. New test cases required:
  - `metric_test.go`: AR with Rollout owner ref → verify `cfg.Namespace/RolloutName/AnalysisRunName/AnalysisRunUID` all populated on the provider call
  - `metric_test.go`: standalone AR (no owner refs) → verify warning path, empty `cfg.RolloutName`, other fields populated
  - `metric_test.go`: user `cfg.Namespace` set in YAML → verify user value wins over `ar.Namespace`
  - `step_test.go`: non-nil Rollout → verify `cfg.Namespace/RolloutName/RolloutUID` populated, AR fields remain empty
  - `step_test.go`: user `cfg.Namespace` set → verify user value wins over `rollout.Namespace`
  - `operator_test.go`: new `TestTriggerRun_WithRolloutUID` → verify Rollout owner ref is set when `AnalysisRunUID` empty but `RolloutUID` present
  - `operator_test.go`: new `TestTriggerRun_AnalysisRunUIDWinsOverRolloutUID` → both UIDs set, verify AR wins
- **D-09:** When the controller passes a nil AR or nil Rollout (should never happen in production, but the gob RPC boundary is worth defending), log a warning and skip metadata extraction. The call continues with whatever `cfg` already holds — user config + the `"default"` safety net. Matches the graceful-degradation pattern from D-04 and keeps pre-fix configurations that happened to work from suddenly breaking.

### Claude's Discretion
- Helper naming and placement for the owner-ref walk (e.g. inline in `Run()/Resume()/Terminate()` vs a `rolloutNameFromAnalysisRun(ar)` helper)
- Log field names and levels for the "nil parent" and "standalone AR" warnings (consistent with existing `slog` usage)
- Test helper signatures for building non-nil AR/Rollout fixtures (e.g. `testAR(opts ...func(*v1alpha1.AnalysisRun))` vs struct literals inline per test)
- Whether to add a single pure-function helper in `provider/` package for the owner-ref selection or keep the branch inline in `operator.go` `TriggerRun`

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `PluginConfig` struct already has `RolloutName`, `AnalysisRunName`, `AnalysisRunUID` fields with `json:"-"` — Phase 8 shaped the surface area, Phase 08.1 populates and extends it
- `mockProvider` pattern in `metric_test.go` / `step_test.go` — tests verify cfg contents by setting `TriggerRunFn`/`GetRunResultFn` closures and asserting on the received `*PluginConfig`
- `operator_test.go` `TestTriggerRun_WithAnalysisRunUID` / `TestTriggerRun_WithoutAnalysisRunUID` — canonical template for the new Rollout-owned owner-ref tests
- Existing `slog` structured logging patterns (JSON to stderr) for the warning paths

### Established Patterns
- Stateless provider interface: all needed data travels via `*PluginConfig` on every call (per Phase 8 D-06). Extending cfg with `RolloutUID` stays on this pattern.
- `json:"-"` tag on injected fields keeps them out of the user-facing config JSON — do the same for `RolloutUID`
- Per-provider validation via `ValidateK6Operator()` in `config.go` — new field is optional, no validation change needed
- Compile-time interface assertion `var _ Provider = (*K6OperatorProvider)(nil)` — no interface change so no adjustment required

### Integration Points
- Three metric methods (`Run`, `Resume`, `Terminate`) each call `parseConfig(metric)` and then operate on `cfg` — metadata extraction goes between those two steps in each method
- Three step methods (`Run`, `Terminate` via `stopActiveRun`, `Abort` via `stopActiveRun`) each call `k.parseConfig(ctx)` — metadata extraction needs to happen in Run (before TriggerRun), and the shared `stopActiveRun` helper needs the Rollout too for StopRun consistency (StopRun takes cfg which may carry the Rollout-owned CR name)
- `operator.go` owner-ref block is a single branch today (`if cfg.AnalysisRunUID != ""`) — refactor to a switch/if-else chain per D-07

</code_context>

<specifics>
## Specific Ideas

- The Argo Rollouts controller creates AnalysisRuns with exactly one controller owner-ref pointing to the Rollout (verified in upstream `rollout/analysis.go:502`). Walking `ar.OwnerReferences` for `kind == "Rollout"` with `Controller == true` is the canonical extraction.
- Phase 8 tests already use `cfg.RolloutName = "my-app"` and `cfg.AnalysisRunUID = "uid-abc-123"` as fixture values — reuse similar values for new tests to stay recognizable.
- After this fix, `e2e/k6_operator_test.go` could drop its explicit `namespace: %s` line from the inline AR YAML (the namespace would flow from AR metadata). **Leaving that cleanup out of scope** — this phase is the fix; the e2e YAML cleanup can come later or be treated as tech debt.

</specifics>

<deferred>
## Deferred Ideas

- Removing the `namespace:` line from e2e test fixtures after the plumbing works (nice-to-have cleanup — not a functional requirement)
- Standalone AnalysisRun support for the step plugin (step plugin conceptually has no AR — out of architectural scope)
- Label-based rollout identification as a fallback when OwnerReferences are missing — explicitly rejected in D-03 in favor of the authoritative owner-ref path

</deferred>
