## Namespace Precedence & Fallback

### Q1: When both user config and parent metadata specify a namespace, which wins?

| Option | Description | Selected |
|--------|-------------|----------|
| User config > Parent > 'default' (Recommended) | Keeps D-07 intent: cfg.Namespace (if user set) wins; else AR/Rollout namespace; else 'default' fallback. Only populate cfg.Namespace from parent when user left it empty. | ✓ |
| Parent namespace always wins | Simpler, predictable: always use AR/Rollout.Namespace, ignore user cfg.Namespace. Removes cross-namespace use case. | |
| User config wins, error if parent missing | Strict: user must explicitly set namespace OR parent must provide one. No 'default' fallback. | |

**User's choice:** User config > Parent > 'default' (Recommended)

### Q2: After this fix plumbs namespace through, what happens to the 'default' fallback in operator.go readScript (line 145–148)?

| Option | Description | Selected |
|--------|-------------|----------|
| Keep as safety net (Recommended) | Leave 'default' fallback so standalone provider use (no parent metadata) still works. The fallback becomes unreachable in normal plugin paths since metric/step populate namespace, but remains defensive. | ✓ |
| Remove fallback — error on empty namespace | After plumbing lands, empty cfg.Namespace is a programming error. Fail fast with a clear message instead of silently using 'default'. | |
| Remove fallback in readScript but keep in TriggerRun | Granular: readScript hard-fails on empty ns (script load is the first operation), TestRun creation can still default. | |

**User's choice:** Keep as safety net (Recommended)

## Rollout Name Derivation (Metric Plugin)

### Q1: How should metric.go extract cfg.RolloutName from the AnalysisRun?

| Option | Description | Selected |
|--------|-------------|----------|
| OwnerReferences (Recommended) | Walk ar.OwnerReferences for kind='Rollout' (Controller=true preferred). This is how the Argo Rollouts controller stamps AR parentage — authoritative and simple. Standalone AR (e2e/manual kubectl apply) will have empty owner refs, so RolloutName stays empty. | ✓ |
| OwnerReferences, fall back to AR.Name | Same as above, but when no Rollout owner ref exists (standalone AR), use ar.Name as the 'rollout' segment of TestRun naming. Pragmatic for e2e tests and manual invocations. | |
| Use labels (e.g. rollouts-pod-template-hash) | AR inherits Rollout.Spec.Selector.MatchLabels. Could extract a label-derived identifier. Less reliable — labels can be arbitrary. | |

**User's choice:** OwnerReferences (Recommended)

### Q2: If no Rollout name can be derived, how should the plugin behave?

| Option | Description | Selected |
|--------|-------------|----------|
| Warn and continue with empty RolloutName (Recommended) | Log a warning, leave cfg.RolloutName empty. operator.go already handles this — TestRun name becomes 'k6-<hash>' without rollout segment. Plugin still works for standalone AR. | ✓ |
| Error — require Rollout ownership | Fail the measurement if no Rollout owner ref found. Forces correct usage but breaks standalone AR scenarios including e2e tests. | |
| Use a static placeholder like 'unknown' | Populate RolloutName='unknown' as a last resort so CR naming remains consistent. Hides the missing-metadata case. | |

**User's choice:** Warn and continue with empty RolloutName (Recommended)

## Step Plugin: AR Fields & TestRun Ownership

### Q1: Step plugin has no AnalysisRun. How should cfg.AnalysisRunName/UID be handled, and what owns the TestRun?

| Option | Description | Selected |
|--------|-------------|----------|
| Leave AR fields empty — no TestRun owner (Recommended) | Minimal change: cfg.AnalysisRunName='', cfg.AnalysisRunUID='' for step plugin. operator.go already skips owner ref when UID is empty. StopRun/Abort/Terminate explicit cleanup remains the primary mechanism. | |
| Extend cfg with RolloutUID, own TestRun by Rollout | Add cfg.RolloutUID field, populate from rollout.UID in step.go. operator.go learns to set owner ref on either AnalysisRun (metric) or Rollout (step). Better GC on rollout deletion but expands interface and changes operator.go scope. | ✓ |
| Reuse AnalysisRunUID field to carry Rollout UID | Repurpose existing field to avoid interface change. Misleading name — tech debt. | |

**User's choice:** Extend cfg with RolloutUID, own TestRun by Rollout
**Notes:** Deliberate scope expansion — accept the `operator.go` owner-ref logic change to get proper GC on Rollout deletion for step-triggered TestRuns.

### Q2: Should step.go populate RolloutName and Namespace from the *v1alpha1.Rollout argument?

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — extract rollout.Name and rollout.Namespace (Recommended) | step.go sets cfg.RolloutName=rollout.Name, cfg.Namespace=(user override || rollout.Namespace || '') before dispatch. Mirrors metric.go plumbing — symmetric fix. | ✓ |
| Only RolloutName; keep namespace user-config-only | Step plugin typically runs inline with Rollout, but namespace stays user-driven. Inconsistent with metric plugin. | |

**User's choice:** Yes — extract rollout.Name and rollout.Namespace (Recommended)

### Q3 (follow-up): When operator.go builds the TestRun OwnerReference, if both AnalysisRunUID and RolloutUID are populated, which wins?

| Option | Description | Selected |
|--------|-------------|----------|
| AnalysisRun wins when present (Recommended) | If cfg.AnalysisRunUID != '' use AnalysisRun owner ref. Else if cfg.RolloutUID != '' use Rollout owner ref. Else no owner ref. Metric plugin naturally picks AR; step plugin picks Rollout. | ✓ |
| Rollout wins when present | Rollout is longer-lived; TestRun outlives AR. Rollout deletion cleans up TestRun. Less responsive to AR abort. | |
| Only one field populated at a time (invariant) | Enforce that metric.go sets only AR fields, step.go sets only Rollout fields. operator.go assumes exactly one. Simpler logic but relies on caller discipline. | |

**User's choice:** AnalysisRun wins when present (Recommended)

## Regression Test Strategy & Error Handling

### Q1: Existing metric/step tests pass nil as the AR/Rollout argument. How should tests verify the new metadata extraction?

| Option | Description | Selected |
|--------|-------------|----------|
| Non-nil AR/Rollout in new tests only (Recommended) | Add targeted tests with real AR/Rollout objects. Keep existing nil-argument tests as-is since they exercise config parsing and don't need metadata. Verify via mockProvider.TriggerRunFn that cfg received the expected fields. | ✓ |
| Update all existing tests to non-nil | Thorough but noisy — hundreds of lines of AR/Rollout fixtures across tests that don't care about metadata. | |
| Helper fixtures + selective non-nil | Extract testAR()/testRollout() helper, apply to new tests. Existing tests keep nil. Middle ground. | |

**User's choice:** Non-nil AR/Rollout in new tests only (Recommended)

### Q2: When the controller passes nil AR or nil Rollout, how should the plugin react?

| Option | Description | Selected |
|--------|-------------|----------|
| Warn-and-continue with empty metadata (Recommended) | Log warning, skip metadata extraction, proceed with empty cfg fields. Falls back to user config + 'default' safety net — matches graceful degradation from Areas 1 and 2. | ✓ |
| Return measurement error | Mark measurement as PhaseError with 'nil AnalysisRun from controller'. Prevents silent misconfig but breaks users whose setups happened to work with the old (broken) discard behavior. | |

**User's choice:** Warn-and-continue with empty metadata (Recommended)
