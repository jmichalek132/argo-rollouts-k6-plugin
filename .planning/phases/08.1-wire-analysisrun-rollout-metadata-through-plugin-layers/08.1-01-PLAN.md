<objective>
Add the `RolloutUID` field to `PluginConfig` and refactor the k6-operator owner-reference helper to encode the D-07 precedence rule: AnalysisRun > Rollout > none. This is the foundation for Phase 08.1 — Wave 2 plans (metric.go + step.go metadata extraction) depend on this field existing and on the owner-ref helper supporting both parent kinds.

Purpose: The step plugin has no AnalysisRun, only a Rollout. Today `buildTestRun`/`buildPrivateLoadZone` call `analysisRunOwnerRef(cfg.AnalysisRunName, cfg.AnalysisRunUID)` which returns nil when AnalysisRunUID is empty — so step-triggered CRs end up with no owner reference and no GC relationship. D-05 adds `RolloutUID`; D-07 says the most-immediate parent wins.

Output: `PluginConfig.RolloutUID` field, a new `parentOwnerRef(cfg)` helper that replaces the two call sites of `analysisRunOwnerRef`, and two new operator tests that prove both the Rollout-owned branch and the AR-wins-over-Rollout precedence.

    // parentOwnerRef returns the OwnerReference chain for a k6-operator CR per D-07.
    // Precedence: AnalysisRun > Rollout > none. The most-immediate parent gets the GC
    // relationship -- metric plugin naturally selects AR; step plugin naturally selects
    // Rollout. When both UIDs are empty, returns nil (no owner reference) which the
    // Kubernetes API server accepts -- CRs will still be discoverable via the
    // app.kubernetes.io/managed-by and k6-plugin/rollout labels.
    func parentOwnerRef(cfg *provider.PluginConfig) []metav1.OwnerReference {
        if cfg.AnalysisRunUID != "" {
            return []metav1.OwnerReference{{
                APIVersion: "argoproj.io/v1alpha1",
                Kind:       "AnalysisRun",
                Name:       cfg.AnalysisRunName,
                UID:        types.UID(cfg.AnalysisRunUID),
            }}
        }
        if cfg.RolloutUID != "" {
            return []metav1.OwnerReference{{
                APIVersion: "argoproj.io/v1alpha1",
                Kind:       "Rollout",
                Name:       cfg.RolloutName,
                UID:        types.UID(cfg.RolloutUID),
            }}
        }
        return nil
    }

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Argo Rollouts controller → plugin (net/rpc gob) | Controller sends PluginConfig (JSON) and AR/Rollout objects over the go-plugin RPC channel. Both sides run inside the same controller pod process group, same service account. Not a classic "untrusted input" boundary, but malformed gob payloads are structurally possible. |
| plugin → kube-apiserver | TestRun/PrivateLoadZone CRs with OwnerReferences are created by the dynamic client. OwnerReferences inform garbage collection; a spoofed UID could potentially redirect GC. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08.1-01 | Tampering | parentOwnerRef helper (testrun.go) | accept | OwnerReferences are set from cfg fields that can only be populated by the plugin layer via AR/Rollout ObjectMeta (Wave 2). The controller is the sole source of these objects and runs as a trusted in-cluster peer. No user-controlled path reaches UID. Risk: LOW. |
| T-08.1-02 | Information Disclosure | PluginConfig JSON marshaling | mitigate | The new `RolloutUID` field uses `json:"-"` tag so it never appears in marshaled JSON sent over logs/metrics/debug output. This matches the pattern already used for `RolloutName`, `AnalysisRunName`, `AnalysisRunUID` (D-05). |
| T-08.1-03 | Denial of Service | decodeRunID / fake client Get in new tests | accept | Tests run against in-memory fake clients. No network or real kube-apiserver. Risk: N/A (test-only). |
| T-08.1-04 | Repudiation | CR OwnerReferences | accept | OwnerReferences are metadata only; they do not grant permissions. Kubernetes RBAC continues to govern who can create/read CRs. The plugin's SA already requires `testruns` create/delete for Phase 8. |
</threat_model>
