## Codex Review

## 08.1-01-PLAN.md

**Summary**
This plan is strong and correctly scoped as the Wave 1 foundation. It cleanly introduces `RolloutUID`, moves owner-ref selection into a single helper, and adds tests for both the new Rollout-owned path and the AR-over-Rollout precedence. It aligns well with D-05, D-07, and D-08, and it preserves the existing no-owner-ref behavior when both UIDs are absent. The main weakness is a small mismatch between the roadmap text and the file list: the roadmap names `internal/provider/operator/operator.go`, while the plan correctly targets `testrun.go`, which could confuse an executor if not made explicit.

**Strengths**
- Clear Wave 1 dependency boundary: this lands the data model and owner-ref logic before metric/step wiring.
- Good adherence to D-07: precedence is explicit and test-backed.
- Correctly preserves existing nil-owner behavior when both UIDs are empty.
- Avoids scope creep: no validation changes, no router/interface churn, no `operator.go` flow edits.
- Test strategy is solid: exercises the full TriggerRun path rather than only unit-testing the helper.
- Security posture is appropriate: `json:"-"` on `RolloutUID` matches the existing injected-field pattern.

**Concerns**
- **MEDIUM**: The plan header says `files_modified` includes `internal/provider/operator/testrun.go`, but the phase overview/canonical refs mention `operator.go`. That inconsistency is understandable from the code layout, but it should be explicitly resolved in the plan text.
- **LOW**: Tests only assert `TestRun` owner refs, while the must-haves also mention `PrivateLoadZone` CRs. The helper is shared, so this is probably enough, but the acceptance criteria overstate direct coverage.
- **LOW**: The helper returns an owner ref when UID is set even if the corresponding name is empty. That mirrors existing behavior, but Kubernetes requires a non-empty owner ref name, so malformed future callers could still create invalid refs.

**Suggestions**
- Add one explicit note: "Although CONTEXT names `operator.go`, the owner-ref helper actually lives in `testrun.go`; that is the intended edit point."
- Either narrow the truth statement from "TestRun and PrivateLoadZone CRs carry..." to "shared helper used by both..." or add one focused test for `buildPrivateLoadZone` if that path is practical.
- Add a brief invariant comment in `parentOwnerRef`: caller is expected to provide both UID and matching name for the chosen parent.

**Risk Assessment**
**LOW**. This is a well-bounded refactor with strong test coverage and correct dependency placement. The main risks are documentation/executor clarity, not design correctness.

## 08.1-02-PLAN.md

**Summary**
This is the strongest of the three plans. It directly addresses the metric-plugin half of the bug by ensuring `AnalysisRun` metadata is injected before provider dispatch in `Run`, `Resume`, and `Terminate`, and it maps closely to D-01, D-03, D-04, D-08, and D-09. The test additions are appropriate and focus on the exact behaviors that were missing. The only notable gap is that the plan does not verify the `Controller == true` preference when multiple Rollout owner refs are present, even though D-03 explicitly calls that out.

**Strengths**
- Precisely fixes the real bug path: `cfg.Namespace`, `cfg.RolloutName`, `cfg.AnalysisRunName`, and `cfg.AnalysisRunUID` are populated before provider calls.
- Good handling of nil AR and standalone AR cases, matching D-04 and D-09.
- Correct namespace precedence: user config wins over AR namespace.
- The helper-based approach avoids repeated logic across `Run`, `Resume`, and `Terminate`.
- Existing nil-AR tests are intentionally preserved, which reduces regression risk.
- Scope is tight: no unrelated changes to parsing, provider interfaces, or e2e YAML.

**Concerns**
- **MEDIUM**: D-03 says "prefer the entry with `Controller == true`." The helper as written returns the first owner ref matching `Kind == "Rollout"` and `Controller == true`, but there is no explicit test for multiple owner refs or for ignoring a non-controller Rollout ref.
- **LOW**: The plan only adds tests for `Run`, not `Resume` or `Terminate`, even though those methods are also being modified. The shared helper lowers risk, but there is still some call-site risk.
- **LOW**: The optional log-field expansion may add noise without much value; it is not wrong, just not necessary for the bug fix.

**Suggestions**
- Add one test fixture with multiple owner refs to confirm the plugin selects the controller Rollout ref and not another ref.
- Add at least one lightweight regression test covering either `Resume` or `Terminate` with a non-nil AR to prove the helper is actually wired into all three paths, not just `Run`.
- Keep the `slog.Info` extension explicitly optional and non-blocking so the executor does not spend time on observability polish.

**Risk Assessment**
**LOW**. The design is tightly aligned with the locked decisions and directly addresses the namespace/metadata bug for the metric path. Remaining risk is mostly around untested call sites and multi-owner-ref edge cases.

## 08.1-03-PLAN.md

**Summary**
This plan is also strong and completes the other half of the bug fix by wiring `Rollout` metadata into the step plugin. It correctly handles D-01, D-05, D-06, D-08, and D-09, and it fixes an important secondary issue: `Terminate` and `Abort` now propagate the `Rollout` through `stopActiveRun`, which is necessary for consistent StopRun behavior. The main weakness is that the tests focus only on `Run`; they do not directly verify that `Terminate` and `Abort` pass a populated config into `StopRun`, which is one of the plan's important claimed outcomes.

**Strengths**
- Correctly recognizes that step has no `AnalysisRun` and uses `Rollout` as the parent.
- Good symmetry with Plan 02 without forcing a fake owner-ref walk where it does not belong.
- Important fix to `stopActiveRun` signature and call sites; this is necessary for `Terminate`/`Abort`.
- Namespace precedence is correct and consistent with the metric side.
- Good attention to import alias collision in tests (`k8stypes`).
- Scope is disciplined: no churn in timeout/state/polling logic.

**Concerns**
- **MEDIUM**: The new tests cover `Run` only. They do not prove that `Terminate` and `Abort` actually deliver populated `cfg.Namespace`, `cfg.RolloutName`, and `cfg.RolloutUID` into `StopRun`, even though that is a key part of the plan objective.
- **LOW**: Acceptance criteria say `grep -c "_ \*v1alpha1.Rollout" ... returns 0`, which is fine now, but brittle if another unrelated helper later adds a blank Rollout arg.
- **LOW**: Nil-Rollout handling is appropriate, but the plan assumes silent continuation is always safe in `stopActiveRun`; if `StopRun` resolution depends on rollout-derived fields and config lacks them, the failure mode remains provider-specific.

**Suggestions**
- Add one targeted `Terminate` or `Abort` test that asserts the `StopRunFn` mock receives populated `cfg.RolloutName`, `cfg.RolloutUID`, and namespace when a non-nil Rollout is supplied.
- Make the `Run + stopActiveRun` wiring the key acceptance point rather than relying mostly on grep counts.
- Add one sentence in the plan clarifying that nil Rollout preserves pre-fix behavior but may still result in fallback provider behavior if required metadata is absent.

**Risk Assessment**
**MEDIUM**. The implementation approach is correct, but the missing direct tests for `Terminate`/`Abort` create a real verification gap around one of the most important step-side changes.

## Overall Assessment

**Summary**
Taken together, the three plans are high quality, tightly scoped, and mostly well aligned with D-01 through D-09. The Wave 1 → Wave 2 dependency ordering is correct: Plan 01 establishes the shared config and owner-ref behavior, then Plans 02 and 03 wire metadata from the plugin layers. The phase goal is largely achieved by these plans: they should fix the namespace propagation bug exposed by Phase 10 e2e and restore correct owner-reference construction. The main gap across the set is verification depth on non-Run call paths: metric `Resume`/`Terminate` and especially step `Terminate`/`Abort` are modified but not tested as directly as they should be.

**Cross-plan strengths**
- Strong decision alignment, especially D-01, D-05, D-07, D-08, D-09.
- Good separation of concerns across waves.
- Good avoidance of scope creep.
- Sensible threat models; nothing materially underestimates the trust boundaries.
- Tests are generally behavior-oriented rather than implementation-only.

**Cross-plan concerns**
- **MEDIUM**: Verification is strongest on creation paths and weaker on stop/resume paths, despite those being part of the bug surface.
- **LOW**: Some acceptance criteria overuse `grep`-style structural checks rather than behavior checks.
- **LOW**: There is mild file-location ambiguity between `operator.go` in the roadmap prose and `testrun.go` in the actual implementation plan.

**Cross-plan suggestions**
- Add one metric test for `Resume` or `Terminate` with a non-nil AR.
- Add one step test for `Terminate` or `Abort` proving `StopRun` receives populated rollout metadata.
- Clarify once, at the top of Plan 01, that the owner-ref helper lives in `testrun.go`, which is the intended implementation target behind the `operator.go` roadmap wording.

**Overall Risk Assessment**
**LOW to MEDIUM**. The design itself is sound and should achieve the phase goal. Risk comes from incomplete verification on some modified call paths rather than from flawed architecture or requirements drift.
