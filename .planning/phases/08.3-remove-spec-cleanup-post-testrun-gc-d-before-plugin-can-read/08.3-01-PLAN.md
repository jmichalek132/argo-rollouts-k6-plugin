<objective>
Remove `spec.Cleanup = "post"` from `buildTestRun`. k6-operator v1.3.x calls
`r.Delete(ctx, k6)` on the TestRun CR (which cascade-deletes runner pods) as
soon as stage transitions to `finished` or `error` when `spec.Cleanup == "post"`.
This race-deletes the CR before the metric plugin's 10s Resume poll can read
the terminal stage and parse `handleSummary` from pod logs, producing
`testruns.k6.io "k6-unknown-..." not found` and driving the AnalysisRun to
`phase=Error` after `consecutiveErrorLimit` retries.

Phase 08.2's `parallelism=1` default is what exposed this bug: before 08.2 the
TestRun sat paused forever with `parallelism=0`, so it never reached a terminal
stage and never triggered the cleanup. 08.2 let TestRuns complete in ~2s, which
surfaced the underlying cleanup race.
</objective>

<root_cause>
Source: `github.com/grafana/k6-operator@v1.3.2/internal/controller/testrun_controller.go:361-366`

```go
case "error", "finished":
    if k6.GetSpec().Cleanup == "post" {
        _ = r.Delete(ctx, k6)
    }
```

The plugin's design requires the TestRun CR to remain observable after completion
so `GetRunResult` can read `stage=finished`, inspect runner exit codes, and parse
`handleSummary` from runner pod logs. `spec.Cleanup=post` defeats that assumption.
</root_cause>

<fix>
1. `internal/provider/operator/testrun.go` — remove `Cleanup: "post"` from the
   `TestRunSpec` literal, add an explanatory Godoc block on `buildTestRun`
   explaining why the field is intentionally unset.
2. `internal/provider/operator/testrun_test.go` — rename existing
   `TestBuildTestRun_Cleanup` (which asserted `Cleanup == "post"`) to
   `TestBuildTestRun_CleanupUnset`, invert the assertion to
   `assert.Empty(string(tr.Spec.Cleanup))`, and add a regression comment
   citing the k6-operator GC semantics.
3. `e2e/k6_operator_test.go` — extend `dumpK6OperatorDiagnostics` to also emit
   AnalysisRun/Rollout yaml and argo-rollouts + k6-operator controller logs on
   failure. The original diagnostic only dumped TestRuns and pods, which showed
   `items: []` at teardown and misled the investigation toward a "no CR ever
   created" theory when in fact CRs were being created and self-deleted within
   seconds of completion.
</fix>

<followup_not_in_scope>
Success-path TestRun cleanup now relies on a manual operator sweep (or is simply
left to accumulate in the namespace). A follow-up phase should:
- Implement `GarbageCollect` on the metric plugin (called by Argo Rollouts after
  terminal analysis) to delete the TestRun CR + pods.
- Add a symmetric post-terminal delete on the step plugin's terminal state
  handler so successful step runs don't leak resources.

Until that phase lands, users running the k6-operator provider should expect
completed TestRun CRs to persist in the target namespace. `StopRun`
(Terminate/Abort paths) still cleans up on cancellation, so the leak only
affects success-path runs.
</followup_not_in_scope>
