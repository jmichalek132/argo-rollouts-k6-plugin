# Phase 08.2: Default cfg.Parallelism=1 when unset - Context

**Gathered:** 2026-04-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Bug fix: `internal/provider/operator/testrun.go:162` forwards `cfg.Parallelism` into `TestRunSpec.Parallelism` with no defaulting. When users omit `parallelism` in their plugin config, the Go zero value `0` is passed to k6-operator, which interprets `spec.Parallelism=0` literally and keeps the TestRun paused — blocking the Phase 10 e2e suite and any real-world deployment that does not explicitly set `parallelism`.

**Fix:** resolve `cfg.Parallelism == 0` to `1` at the TestRun construction site so the emitted CR carries `spec.Parallelism=1`. Validation layer semantics are preserved (0 remains syntactically valid at the config boundary — it means "unset, apply default").

**Scope is confined to:**
- `internal/provider/operator/testrun.go` — `buildTestRun` default application and Godoc
- `internal/provider/operator/testrun_test.go` — two new builder tests
- `internal/provider/config.go` — Godoc on `PluginConfig.Parallelism`
- `examples/k6-operator/README.md` — one-line note about the default
- `e2e/k6_operator_test.go` (or a new sub-test) — regression guard exercising parallelism-unset

**Out of scope:**
- Changing `ValidateK6Operator()` behavior (the `Parallelism < 0` rejection and the "0 is valid" allowance both stay)
- Adding defaults for any other `PluginConfig` field
- PrivateLoadZone construction (does not consume `cfg.Parallelism`)
- Changing the `parseConfig` layer or mutating `cfg` during parse

</domain>

<decisions>
## Implementation Decisions

### Default application site
- **D-01:** Resolve `cfg.Parallelism == 0 → 1` **inside `buildTestRun`** at the TestRun construction site (`internal/provider/operator/testrun.go`). Concretely: compute a local `p := int32(cfg.Parallelism); if p == 0 { p = 1 }` before assigning to `tr.Spec.Parallelism`. Do NOT introduce a new `ResolvedParallelism()` helper on `PluginConfig` (premature abstraction — no second caller exists). Do NOT mutate `cfg.Parallelism` in `parseConfig` (that breaks the `Parallelism: 0 means unset` invariant asserted by the existing `TestValidateK6Operator_ZeroParallelism` test, which remains authoritative per D-03 below).

### Observability
- **D-02:** **Silent default — no log emission** in `buildTestRun`. Keep the function a pure builder (no logger dependency, no side effects, matches the existing D-06 "stateless/pure" convention carried through from Phase 8). Operators can verify the applied parallelism by inspecting the emitted TestRun CR directly (`kubectl get testrun <name> -o yaml`) and via the new builder tests. If a future incident argues for it, adding logging later is mechanical.

### Validation layer (unchanged)
- **D-03:** `ValidateK6Operator()` is **NOT modified**. It continues to:
  - reject `Parallelism < 0` with the existing error message `"parallelism must be non-negative, got %d"`
  - accept `Parallelism == 0` as valid ("means unset, default applied downstream")
  - accept `Parallelism > 0` as-is
  The existing `TestValidateK6Operator_ZeroParallelism` and `TestValidateK6Operator_NegativeParallelism` both stay as-is. The default is a builder-layer concern, not a validation-layer concern.

### Test coverage
- **D-04:** Add two new builder tests in `internal/provider/operator/testrun_test.go` (sibling to the existing `TestBuildTestRun_Parallelism` at line 89):
  - `TestBuildTestRun_DefaultsParallelismWhenUnset` — cfg.Parallelism unset → `tr.Spec.Parallelism == int32(1)`
  - `TestBuildTestRun_PreservesExplicitParallelism` — cfg.Parallelism=4 → `tr.Spec.Parallelism == int32(4)` (pins "only zero defaults" after the change; partial overlap with the existing `TestBuildTestRun_Parallelism` is intentional — this one documents the non-overriding rule)
- **D-05:** Keep the existing tests unchanged:
  - `TestValidateK6Operator_ZeroParallelism:376` — validation still says 0 is valid
  - `TestValidateK6Operator_NegativeParallelism:362` — validation still rejects negatives
  - `TestBuildTestRun_Parallelism:89` — explicit 4 survives (this becomes redundant with D-04's second test; leave as-is rather than deleting — the test cost is negligible)
- **D-06:** Add an **e2e regression guard** to unblock Phase 10. Extend `e2e/k6_operator_test.go` (or add a new scenario function) that runs the full AnalysisRun → TestRun flow with `parallelism` **omitted** from the plugin config and asserts the TestRun reaches a terminal non-paused state. This is the authoritative verification that the bug is gone in the real kind-cluster path, not just in unit builders.

### Docs
- **D-07:** Update Godoc on `PluginConfig.Parallelism` in `internal/provider/config.go:34` to: `// Parallelism is the number of k6 runner pods. Defaults to 1 when unset or zero.` Also add a sentence to the `buildTestRun` Godoc noting the default application.
- **D-08:** Add a one-line note to `examples/k6-operator/README.md` under a "Parallelism" subsection (or, if no subsection exists, append to the nearest config-reference section): `If parallelism is omitted, the plugin submits the TestRun with parallelism=1.` Do NOT add `parallelism: 1` to the example AnalysisTemplate YAML — the whole point of the default is that users shouldn't need to set it.

### Claude's Discretion
- The exact wording of the new test function comments
- Whether the `p := int32(cfg.Parallelism); if p == 0 { p = 1 }` block uses a local var, a small helper in the same file (`defaultParallelism(int) int32`), or an inline ternary-style expression — all three are acceptable
- README.md subsection placement / heading style, as long as the one-line note appears
- Whether the e2e regression uses a new test function or parameterizes an existing one — decide based on whichever keeps the e2e suite readable

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `buildTestRun(cfg, scriptCMName, scriptKey, namespace, crName)` signature (testrun.go:140) — already takes everything it needs; no signature change required
- `TestBuildTestRun_Parallelism` (testrun_test.go:89) — canonical shape for the two new builder tests (build cfg, call `buildTestRun`, assert on `tr.Spec.Parallelism`)
- `TestValidateK6Operator_ZeroParallelism` (testrun_test.go:376) — pins the "0 is valid at validation" rule; stays as-is and becomes the pair to the new builder test

### Established Patterns
- Builder functions in `testrun.go` are pure: no I/O, no logging, no cfg mutation. D-02 preserves this.
- `PluginConfig` carries user config + injected fields; defaulting is done at the consumer site, not in the config struct. D-01 preserves this.
- Tests go next to their subject function (builder tests next to `TestBuildTestRun_*`; validation tests next to `TestValidateK6Operator_*`). D-04/D-05 preserve this.
- Existing tests use `require.Equal(t, int32(N), tr.Spec.Parallelism)` — keep this assertion style.

### Integration Points
- `buildTestRun` is called from `TriggerRun` in `internal/provider/operator/operator.go` — no caller-side change needed
- `cfg.Parallelism` is only read in two places: `ValidateK6Operator` (config.go:89) and `buildTestRun` (testrun.go:162). Fix-site is unambiguous.
- e2e path: plugin RPC → metric/step layer `parseConfig` → router → `K6OperatorProvider.TriggerRun` → `buildTestRun` → `client.Create(testRun)`. The defaulting lands at the last internal step before the API server sees the CR.

</code_context>

<specifics>
## Specific Ideas

- The existing `TestValidateK6Operator_ZeroParallelism` explicitly says in its assertion message `"parallelism 0 means unset, should be valid"` — this comment was prophetic and its contract must not be broken by this phase.
- The new test name format should mirror the existing house style: `TestBuildTestRun_DefaultsParallelismWhenUnset` and `TestBuildTestRun_PreservesExplicitParallelism` both follow the `TestBuildTestRun_<Behavior>` pattern used throughout testrun_test.go.
- Phase 08.1 set a precedent that decimal-phase bug fixes stay narrow: one conceptual fix, existing tests preserved, new tests added alongside. This phase continues that style.

</specifics>

<deferred>
## Deferred Ideas

- Adding a structured-log warning when the default is applied (explicitly declined in D-02 — revisit only if operators report confusion)
- A `ResolvedParallelism()` helper on `PluginConfig` (declined in D-01 — premature abstraction; revisit if a second consumer appears, e.g., a future provider or a metrics exporter)
- Setting an explicit `parallelism: 1` in `examples/k6-operator/` YAML (declined in D-08 — defeats the purpose of the default; the README note carries the information instead)
- Centralizing all PluginConfig defaults in a single `ApplyDefaults()` method (no other field currently needs defaulting; keep the fix surgical)
- Changing k6-operator upstream to default `Parallelism=1` itself (out of this project's scope; the plugin must cope with k6-operator as shipped)

</deferred>
