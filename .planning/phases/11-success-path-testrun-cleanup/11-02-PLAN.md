<objective>
Add the symmetric success-path cleanup hook to the step plugin: when `provider.GetRunResult` returns a terminal state (`Passed`/`Failed`/`Errored`), `K6StepPlugin.Run` fires `provider.Cleanup` once, guarded by a new `state.CleanupDone` flag that persists across Run() invocations via the serialized `ctx.Status`. Aborted is excluded (Terminate/Abort already handled this path via StopRun); Running is obviously excluded; timeout is excluded (StopRun already fires in the timeout branch).

Purpose: Parity with the metric plugin's GarbageCollect hook (11-01) for the step plugin lifecycle. Argo Rollouts calls `Run()` repeatedly until the step reaches a terminal phase; the FIRST terminal observation is the natural hook point for cleanup, and the `CleanupDone` guard ensures we never fire twice (controller reconciliation races can deliver subsequent Run() calls even after a terminal result — we must skip them). This closes the success-path leak for step-plugin-driven TestRun CRs that `spec.Cleanup=post` removal in Phase 08.3 opened up.

Output: `stepState` gains a `CleanupDone bool` field with JSON tag `cleanupDone,omitempty`. `Run()` terminal switch gains a pre-switch guard block that invokes `k.provider.Cleanup(ctx, cfg, state.RunID)` exactly once per terminal transition, logs errors at Warn, and swallows them per GC-03. Seven new unit tests pin the behavior. Two files modified.
</objective>

// Success-path cleanup hook (GC-02).
// Fires on the first observation of a success-path terminal state
// (Passed, Failed, Errored) and NEVER on Running or Aborted:
//   - Running: run still in flight
//   - Aborted: Terminate/Abort RPC already called StopRun, which on the

// Guarded by state.CleanupDone for once-per-terminal-transition semantics.
isSuccessPathTerminal := result.State == provider.Passed ||
    result.State == provider.Failed ||
    result.State == provider.Errored
if isSuccessPathTerminal && !state.CleanupDone && state.RunID != "" {
    cleanupCtx, cleanupCancel := context.WithTimeout(context.Background(), providerCallTimeout)
    if cleanupErr := k.provider.Cleanup(cleanupCtx, cfg, state.RunID); cleanupErr != nil {
        slog.Warn("failed to cleanup run after terminal state",
            "runId", state.RunID,
            "rollout", cfg.RolloutName,
            "namespace", cfg.Namespace,
            "state", string(result.State),
            "error", cleanupErr,
        )
    }
    cleanupCancel()
    state.CleanupDone = true
}

**Design decisions (encoded here so executor does not revisit them):**

- **Hook location: inside `Run()` post-switch, NOT in a separate method.** `Run()` is the natural entry point where Argo Rollouts observes state transitions. We fire cleanup on the same call that sets `FinalStatus`, guarded by `CleanupDone` so subsequent Run() calls (race replay) skip it.

- **Guard mechanism: new `CleanupDone bool` field, NOT reuse `FinalStatus != ""`.** Reusing `FinalStatus` would conflate "terminal seen" with "cleanup fired." The semantics differ: `FinalStatus` is set as soon as the switch matches a terminal state — which on the SAME Run() call happens BEFORE cleanup fires. So checking `FinalStatus != ""` inside the same call would skip cleanup on its very first opportunity. A distinct `CleanupDone` field fires cleanup on the first terminal observation and suppresses it on subsequent replays. JSON tag `cleanupDone,omitempty` keeps serialized state compact for existing runs in flight during the v0.4.0 upgrade (backward compatible: absent field deserializes as `false`, which is the correct default).

- **Terminal states that trigger cleanup: Passed, Failed, Errored.** Aborted is excluded because it means `Terminate`/`Abort` was called (the RPC side-channel for user-initiated cancellation), which already invoked `StopRun` via `stopActiveRun`. On the operator backend, StopRun and Cleanup share `deleteCR` — firing Cleanup again would be a redundant NotFound roundtrip. On the grafana-cloud backend, both are no-ops anyway. Cleaner to keep the explicit exclusion than to rely on NotFound-dedup.

- **Timeout branch unchanged.** The timeout path (step.go:134-151) already calls StopRun and returns early with PhaseFailed. Adding Cleanup there would be redundant (StopRun already deleted the CR via deleteCR). The timeout branch sets `FinalStatus="TimedOut"` — which is intentionally NOT in the `isSuccessPathTerminal` set.

- **Best-effort: CleanupDone is set to true REGARDLESS of Cleanup result.** Per GC-03, cleanup errors don't fail the step. Per REQUIREMENTS.md Out of Scope `Retry loop on cleanup failure`, we do not retry. Setting `CleanupDone = true` after a failed Cleanup call means the next Run() (if any) will skip. This is the correct semantics: "we tried, it failed, move on."

- **Context: fresh `context.Background()` per Cleanup call, not derived from an ambient ctx.** The step plugin's `Run()` has no incoming context (types.RpcStepContext is not a context.Context). Mirrors the pattern used for TriggerRun/GetRunResult/StopRun in the existing code.

- **No change to stopActiveRun (Terminate/Abort helper).** Cancellation paths are already correct — they call StopRun which, post-11-01, uses the shared deleteCR helper. Adding Cleanup to stopActiveRun would be a second delete on the same CR (NotFound-safe but wasteful and noisy in logs).

- **Two-file scope is intentional.** The step plugin changes live entirely in `internal/step/`. Provider interface and operator/cloud implementations are 11-01's concern. E2E verification is Phase 12's concern (TEST-02 combined-canary will exercise BOTH 11-01 and 11-02 paths in a real cluster).
