3. Create `e2e/mock_k6_server.go` with `//go:build e2e` tag. The mock must handle these endpoints matching the GrafanaCloudProvider's actual HTTP calls:

   - `POST /loadtests/v2/tests/{id}/start-testrun` -- returns `{"id": <runId>}` with status 200. The runId is a configurable value.
   - `GET /loadtests/v2/test_runs/{id}` -- returns a JSON body with `id`, `status_text`, `result_text`, `url` fields. The response is pulled from a per-test sequence: first N calls return `{"status_text": "running", "result_text": null}`, then terminal call returns `{"status_text": "finished", "result_text": "passed"}` or `"failed"`.
   - `POST /loadtests/v2/test_runs/{id}/stop` -- returns 200.
   - `GET /cloud/v5/test_runs/{id}/query_aggregate_k6` -- returns aggregate metrics JSON with `data` array containing metric objects for `http_req_failed`, `http_req_duration`, `http_reqs`. Values are configurable per test.

   The mock server uses `sync.Mutex` for thread safety. Each test programs its own response sequences via `OnTriggerRun(runId)`, `OnGetRunResult(runId, responses...)`, and `OnAggregateMetrics(runId, metrics)` methods.

   IMPORTANT: The GrafanaCloudProvider constructs its base URL from `stackId` as `https://<stackId>.grafana-dev.net`. For e2e tests, the PluginConfig `stackId` field CANNOT encode a localhost URL this way. Instead, the provider's `WithBaseURL` functional option (already in cloud.go) overrides the base URL. However, the plugin binary uses `cloud.NewGrafanaCloudProvider()` without WithBaseURL. The solution: the mock server must be reachable from inside the kind cluster. The PluginConfig in AnalysisTemplate YAML will have `stackId` set to a dummy value, and the `apiToken` is just `"test-token"`. The mock server listens on the host; from inside the kind node, the host is reachable via Docker's host gateway.

   ALTERNATIVE approach (simpler, recommended): Instead of trying to reach the mock from inside the kind cluster (complex networking), use a sidecar mock deployment inside the cluster. Deploy a simple Go HTTP mock as a Kubernetes Pod + Service in the test namespace, and point the plugin config at that service URL. BUT this requires the provider to accept an arbitrary base URL in config, which it currently does via the stackId-based URL construction. We need to make the GrafanaCloudProvider's base URL overridable via config or env var.

   SIMPLEST approach: Set an environment variable `K6_BASE_URL` that the GrafanaCloudProvider reads. If set, it overrides the stackId-based URL. Add this to cloud.go as a one-line check: `if baseURL := os.Getenv("K6_BASE_URL"); baseURL != "" { ... }`. Then in the kind cluster, set this env var on the argo-rollouts controller deployment (or via the plugin process environment via ConfigMap). Since the plugin binary is a child process of the controller, env vars from the controller pod propagate to plugin processes.

   NOTE: The executor should evaluate the best approach for making the mock server reachable from inside kind. The three options above are listed in order of complexity. The key constraint is that GrafanaCloudProvider constructs `https://<stackId>.grafana-dev.net` as the base URL -- the mock must be reachable at whatever URL the provider constructs or be configured to use a different URL.

   TestMain lifecycle:
   a. Create kind cluster with random name (`envconf.RandomName("k6-plugin", 16)`)
   b. Create test namespace
   c. Install Argo Rollouts: `kubectl apply -n argo-rollouts -f https://github.com/argoproj/argo-rollouts/releases/download/v1.9.0/install.yaml` (this installs CRDs + controller)
   d. Cross-compile both plugin binaries for linux/amd64: `GOOS=linux GOARCH=amd64 CGO_ENABLED=0 go build -o /tmp/e2e-plugins/metric-plugin ./cmd/metric-plugin` and same for step-plugin
   e. Copy binaries into kind node using `docker cp <binary> <kind-node>:/tmp/argo-rollouts/`
   f. Apply argo-rollouts-config ConfigMap with `file:///tmp/argo-rollouts/metric-plugin` and `file:///tmp/argo-rollouts/step-plugin` paths
   g. Restart the argo-rollouts controller to pick up the plugin config
   h. Teardown: delete namespace, destroy kind cluster
