**3. examples/canary-full/ (EXAM-03)**

CRITICAL: This example MUST NOT show testRunId handoff. Per research Pitfall 7 and key_research_findings #3, the step plugin and metric plugin trigger INDEPENDENT k6 test runs using the same testId. There is no `{{steps.X.outputs.Y}}` expression in Argo Rollouts v1.9.0.

`rollout.yaml`:
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: canary-with-k6
spec:
  replicas: 3
  revisionHistoryLimit: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: nginx:1.27
          ports:
            - containerPort: 80
  strategy:
    canary:
      steps:
        # Step 1: Route 20% traffic to canary
        - setWeight: 20
        # Step 2: Run k6 load test as a gate (trigger + wait for pass/fail)
        - plugin:
            name: jmichalek132/k6-step
            config:
              testId: "<YOUR_TEST_ID>"
              apiToken: "<YOUR_API_TOKEN>"
              stackId: "<YOUR_STACK_ID>"
              timeout: "10m"
        # Step 3: Run metric analysis (triggers its own independent k6 run)
        - analysis:
            templates:
              - templateName: k6-threshold-check
            args:
              - name: api-token
                valueFrom:
                  secretKeyRef:
                    name: k6-cloud-credentials
                    key: apiToken
              - name: stack-id
                valueFrom:
                  secretKeyRef:
                    name: k6-cloud-credentials
                    key: stackId
              - name: test-id
                value: "<YOUR_TEST_ID>"
        # Step 4: If both gates pass, promote to 50%
        - setWeight: 50
        - pause: {duration: 30s}
        # Step 5: Full promotion
        - setWeight: 100
```

NOTE about the step plugin config: The step plugin reads `apiToken` and `stackId` directly from the config JSON (not from secretKeyRef), so these are inline placeholder values. For production, users should use a Kubernetes Secret and reference it. Add a YAML comment explaining this.
